## Herramientas del Proyecto

El equipo utiliza un stack de herramientas moderno para gestionar el desarrollo ágil, el diseño de interfaces y el control de versiones del código.

| Herramienta | Propósito | Uso en el proyecto |
|-------------|-----------|-------------------|
| **Jira** | Gestión ágil del trabajo | Planificación de épicas, historias de usuario, subtareas y seguimiento de sprints bajo metodología Scrum. |
| **Figma** | Diseño de prototipos y UI | Creación de mockups de alta fidelidad, wireframes estructurales, flujos de navegación y validación de heurísticas UX/UI. |
| **GitHub** | Control de versiones | Repositorio compartido del código fuente, flujo colaborativo con ramas, pull requests y revisión entre pares. |
| **Canva** | Assets visuales y documentación | Apoyo en la creación de identidad visual, piezas de presentación y materiales de documentación gráfica. |
| **Google Drive** | Almacenamiento compartido | Carpeta centralizada para documentos, recursos y respaldos del equipo. |

## Canales de Comunicación

| Canal | Propósito | Frecuencia |
|-------|-----------|------------|
| **Jira** | Seguimiento de tareas, asignación de responsables y reporte de estados | Diario / Cada Sprint |
| **GitHub** | Gestión de código, revisiones de pull requests y versionado | Continuo |
| **Reuniones de sincronización** | Daily Scrum y planificación de sprint | Diario / Semanal |

## Acuerdos del Equipo

- **Convención de commits:** Se siguen prácticas de commits semánticos para mantener un historial claro (`feat:`, `fix:`, `docs:`, `chore:`).
- **Convención de ramas:** Uso de ramas por funcionalidad (`feature/descripcion-corta`) integradas a través de pull requests.
- **Revisiones de código:** Todo cambio significativo requiere al menos una revisión por parte de otro miembro del equipo antes de fusionarse a la rama principal.
